# Zerlum 景观设计工作台重构设计

日期：2026-07-10
状态：待用户书面审阅

## 1. 背景

当前产品是以照明设计为领域内核的 React + Vite 工作台。前端、开发服务器、线上 API、Skill、Markdown、知识索引、项目类型、岗位权限、登录后引导和方案输出中均存在照明设计规则。产品已有三个可复用的核心工作面：Agent 对话、AI 无限画布和方案文本。

本次重构的目标不是更换少量文案，而是把产品转为单人使用的综合景观设计工作台。三个工作面继续保留，但必须围绕景观设计师从场地理解到方案交付的真实工作流重新组织。

## 2. 已确认目标

1. 产品领域从照明设计完整切换为景观设计。
2. 保留三个核心工作面：景观 Agent、方案画布、文本交付。
3. 重建后端 Skill、Markdown、提示词和知识运行说明。
4. 欢迎页、登录页和工作台使用同一视觉系统。
5. 取消多人协同、项目码、成员、岗位和权限流程。
6. 用户登录或注册后直接进入工作台。
7. 优化全部页面的信息层级、空状态和操作路径。
8. 解决无限画布节点右侧连接器难以命中的问题。
9. 交付结果应可构建、可测试、可在本地完整体验，并为对外部署保留清晰的生产边界。

## 3. 产品定位

Zerlum 是单人使用的综合景观设计工作台，默认覆盖居住、商业、公园、滨水、酒店、校园和文旅等项目类型。它帮助景观设计师整理项目简报、理解场地、形成设计方向、推演关键节点、生成视觉成果并组织汇报文档。

核心工作流：

```text
欢迎与登录
  → 自动进入当前景观项目
  → Agent 建立和补全项目简报
  → 画布组织场地资料并推演方案
  → 文本交付生成大纲、页面和导出结果
  → 返回 Agent 或画布继续迭代
```

三个工作面共享同一个项目上下文。用户不需要在不同页面重复上传资料或重新解释项目背景。

## 4. 领域语言

产品统一使用根目录 `CONTEXT.md` 中的词汇：景观项目、项目简报、场地资料、设计方向、设计节点、方案成果、交付文档、设计依据和待复核项。

用户界面、前端类型、API 请求、提示词和测试使用同一组概念。不得继续使用照明项目、项目空间、协同空间、灯光节点或 Lighting Skill 等旧词。

## 5. 项目上下文

每个景观项目包含：

- 项目名称
- 项目类型
- 项目地点
- 设计阶段
- 客户或委托方
- 项目目标
- 使用人群
- 场地范围和已知限制
- 场地资料
- 已确认的设计方向
- 方案成果
- 交付文档状态

首次登录时自动建立“未命名景观项目”。用户直接进入景观 Agent，可在工作台内逐步补全项目内容，不再经过创建团队、加入项目码或选择岗位的引导。

## 6. 工作面设计

### 6.1 景观 Agent

Agent 是默认入口，也是项目上下文的解释层。

布局：

- 左侧为项目简报和场地资料，可折叠。
- 中间为主对话区。
- 右侧为最近形成的设计方向、设计节点和待确认事项，可折叠。

空状态提供与景观流程对应的快捷任务：

- 分析场地问题与机会
- 提出概念方向
- 梳理功能与游线
- 深化关键设计节点
- 建立植物与季相策略
- 检查方案完整性

Agent 回答应区分项目事实、设计判断和待复核项。缺少关键信息时，先说明假设，并给出最少且具体的补充问题。

### 6.2 方案画布

画布用于组织场地底图、现状照片、意向参考、生成图、局部深化和可选的空间漫游视频。

空画布直接提供：

- 上传场地底图
- 上传现状照片
- 添加意向参考
- 创建首个方案图节点

节点操作按景观设计任务表达：

- 生成方案图
- 添加参考
- 创建方向变体
- 局部深化
- 调整植物、材料、季节或使用场景
- 生成漫游视频

图像之间的关系使用明确角色：

- 场地底图
- 主场景
- 风格参考
- 材料参考
- 植物参考
- 视频首帧
- 视频尾帧

生图默认保持原图视角、透视、尺度和未要求改变的主体。只有用户明确选择概念改造或自由生成时，模型才可改变空间结构。

### 6.3 文本交付

文本工作面使用清晰的四步流程：

```text
资料确认 → 大纲生成 → 页面生成 → 校对与导出
```

默认方案结构根据项目资料动态选择，可包含：

- 项目理解与目标
- 场地问题和机会
- 设计概念与核心命题
- 总体空间结构
- 功能、游线与使用场景
- 关键节点
- 竖向、排水与海绵策略
- 植物与季相策略
- 材料、铺装和构筑物
- 生态、运营和分期
- 景观夜间体验
- 总结与下一步

画布生成图只用于封面、核心方向、重点节点、对比或体验展示。分析页优先使用结构图、流程图、材料板、植物板和有依据的项目内容。

## 7. 欢迎与登录体验

欢迎页、登录弹窗和工作台使用同一套色彩、排版和动效语言。

欢迎页内容保持克制：Zerlum 品牌标识、主张“从场地到方案”和明确的登录入口。当前独立的高强度棱镜视觉被收敛为与工作台一致的石墨色空间背景。

登录成功后的过渡：

1. 品牌标识保持连续并归位到工作台顶栏。
2. 三个核心工具导航出现。
3. 内容区进入景观 Agent。
4. 不显示任何多人协同或二次入门弹窗。

动效使用短时长缓出曲线，并提供 `prefers-reduced-motion` 替代。

## 8. 视觉系统

使用适合设计师长时间工作的低眩光界面：

- 深色主题以中性石墨黑为主体。
- 浅色主题使用冷白和灰绿中性色。
- 低饱和苔绿只用于当前状态、确认和关键操作，不把整个界面染成“园林绿”。
- 采用清晰的 8px 间距节奏和 12–16px 卡片圆角。
- 减少装饰性玻璃、宽软阴影和无功能的大面积空白。
- 控件标签使用动作和对象，如“生成方案图”“补充场地资料”“导出方案”。
- 保证键盘焦点、文本对比度、触控命中区和窄屏可用性。

工作台顶栏包含品牌、当前项目、设计阶段、三个核心工具、主题和账户。项目补充和切换不遮挡核心任务。

## 9. Landscape Design Skill

删除 `api/zerlum-lighting-skill`，建立 `api/zerlum-landscape-skill`。Skill 由一个入口和以下参考 Markdown 组成：

1. 景观设计通用思维与任务判断
2. 场地解读、问题与机会
3. 空间结构、功能布局与游线
4. 使用人群、场景和运营
5. 竖向、排水、海绵和生态
6. 植物群落、季相和地域适应性
7. 材料、铺装、构筑物和细部
8. 项目类型策略
9. 景观效果图和视频提示词
10. 汇报方案结构
11. 输出质量标准与差异化变量

Skill 是设计判断框架，不是固定风格模板，也不是项目事实来源。它不能覆盖用户要求或场地资料。

## 10. 模型约束

所有模型调用遵循：

```text
用户明确要求 > 项目资料 > 画布显式关系 > 已确认结论 > Landscape Skill > 通用模型知识
```

### 10.1 Agent

先判断任务属于项目简报、场地分析、概念方向、空间结构、功能游线、节点深化、植物、材料、生态、表达或方案评审。回答必须说明依据、假设和需要复核的内容。

不得虚构红线、高程、气候、植物适生性、规范条文、工程量或造价结论。

### 10.2 生图

先判断任务类型：

- 保结构优化
- 概念改造
- 局部替换
- 方向变体
- 季节或时间变化
- 自由生成

提示词必须控制空间层次、植物成熟度、材料真实性、人尺度、天气、季节和使用场景。不得默认蓝调夜景，不得无依据增加路径、水景、构筑物或地形。

### 10.3 视频

保持主体和空间连续性，描述镜头路径、速度、视线高度、关键帧变化、植被运动和人群活动。视频只作为画布中的可选表达，不成为第四个主工作面。

### 10.4 方案文本

只使用用户提交资料、Agent 已形成的项目结论和画布中的方案成果。Skill 提供结构和设计方法，但不能被描述为项目依据。

## 11. 知识与运行说明迁移

删除照明知识索引和照明标准文档，建立景观知识目录和新的运行说明。首轮只部署经过编写和检查的 Landscape Skill Markdown，不虚构外部案例或规范正文。

知识材料必须保留来源、适用项目类型、知识等级、隐私级别和复核状态。涉及规范、无障碍、消防、结构、竖向、排水或植物适生性的结论必须标为待专业复核，除非有明确且有效的项目依据。

本地开发和线上 API 必须引用同一个 Skill 加载器和提示词构建器。不得在 `vite.config.ts` 和 `api/_zerlum-server.ts` 中维护两套不同的领域文案。

## 12. 状态与迁移

新状态只保留：

- 主题
- 单人会话
- 景观项目列表
- 当前项目
- 项目资料
- 三个工作面的必要状态

删除：

- Collaboration
- TeamMember
- MemberRole
- Permissions
- 项目码
- 创建、加入和个人创作分支

使用新的景观工作台存储版本，防止旧照明数据自动注入新提示词。可迁移主题和基础账户显示信息，旧项目与旧聊天不自动转为景观项目。

账号层通过小型适配边界与工作台分离。开发环境可以使用本地会话；面向真实公网账号时必须接入服务器端身份服务，客户端本地状态不能被描述为安全认证。

## 13. 组件边界

当前大型 `App.tsx` 按职责拆分：

- `src/domain`：领域类型、默认值和纯函数
- `src/state`：会话、项目和持久化
- `src/shell`：欢迎、登录、顶栏和项目上下文
- `src/views/agent`：Agent 工作面
- `src/views/canvas`：画布、节点、连线和生成状态
- `src/views/document`：大纲、页面和导出
- `src/components`：无领域状态的通用控件
- `api/zerlum-landscape-skill`：领域 Markdown
- `api/zerlum-landscape-skill.js`：Skill 加载和提示词封装

每个工作面通过明确接口读取同一个项目上下文，不直接依赖其他工作面的内部状态。

## 14. 画布连接器根因与设计

现有连接器按钮为 32×32px。显示状态把按钮从节点边缘继续向右平移，按钮最终与节点之间形成约 2px 的无响应缝隙。指针跨过该缝隙时，节点失去悬停，连接器恢复 `pointer-events: none`，因此出现间歇性难以点击。

新连接器要求：

- 命中区至少 48×48px。
- 命中区与节点边缘重叠，不移动交互盒本身。
- 只缩放内部图标，不改变命中区几何位置。
- 节点悬停、选中、连接器悬停、拖拽和菜单打开期间持续可用。
- 单击打开景观任务菜单，拖拽创建连线。
- 使用指针捕获，拖拽时显示草稿连线和吸附目标。
- 支持键盘焦点和清晰的焦点样式。
- 为命中区、持续可交互状态和指针坐标添加回归测试。

## 15. 错误处理

- 资料不足：说明缺少内容，保留输入，并允许带假设继续。
- 不支持的文件：显示文件名、原因和允许格式。
- API 密钥缺失：显示配置错误，不伪装为生成失败。
- Agent 流中断：保留已接收文本和重试入口。
- 生图或视频失败：只影响当前节点和当前版本。
- 方案页失败：支持单页重试，已完成页面不丢失。
- 浏览器持久化超限：提示用户保存或清理大型资料，不静默丢弃。
- 导出失败：保留页面内容并显示可操作的原因。

## 16. 测试策略

### 16.1 领域和提示词测试

- Skill 文件全部可加载。
- 前端和后端使用景观领域词汇。
- Agent、生图、视频和方案文本分别包含对应景观约束。
- 用户要求和项目资料优先于 Skill。
- 照明 Skill 路径和照明身份提示不再存在。

### 16.2 交互测试

- 登录提交后直接进入 Agent。
- 不出现多人引导、项目码、成员或岗位入口。
- 三个核心工具可切换并共享项目资料。
- 项目简报可补充和持久化。
- 连接器单击、拖拽、菜单持续状态和键盘焦点可用。
- 画布节点失败不破坏其他节点。
- 方案页面可逐页生成和重试。

### 16.3 视觉与可用性验证

- 欢迎、登录和工作台连续。
- Agent、画布和文本交付的空状态可理解。
- 桌面、窄屏和减少动效模式可用。
- 页面无横向溢出、遮挡、裁切菜单或不可见焦点。
- 浏览器控制台无运行时错误。

### 16.4 工程验证

- 全量测试通过。
- TypeScript 检查通过。
- 生产构建通过。
- 本地 HTTP 服务返回成功。
- 本地浏览器完成关键流程验证。

## 17. 完成标准

只有以下证据全部成立，重构才可视为完成：

1. 源码、API、Skill、Markdown、知识目录和用户界面已切换到景观设计逻辑。
2. 遗留照明领域文本审计无未解释命中；第三方包名中的 `light` 不属于领域残留。
3. 用户登录后直接进入可使用的景观 Agent。
4. 三个工作面形成可往返的项目设计闭环。
5. 画布连接器问题有失败测试、修复和交互验证。
6. 视觉系统符合简洁、低眩光、专业和高级的目标。
7. 错误和空状态提供明确下一步。
8. 全量自动化验证和浏览器关键流程验证通过。

## 18. 非目标

- 不恢复多人协同、组织管理或角色权限。
- 不新增第四个主工作面。
- 不把 CAD、GIS 或 BIM 原生编辑器塞入当前版本。
- 不用模型输出替代测绘、结构、排水、植物、造价或规范专业复核。
- 不把客户端本地会话宣称为生产级身份认证。

## 19. 实施顺序

1. 建立领域模型、Landscape Skill 和共享提示词构建器。
2. 删除多人状态并完成登录直达工作台。
3. 重构工作台框架和三个工作面的空状态。
4. 重构画布领域关系并修复连接器。
5. 重构方案文本结构、错误处理和导出状态。
6. 迁移知识与运行说明，完成遗留照明审计。
7. 完成自动化测试、生产构建和浏览器验收。
